# AGENTS.md

## 开发原则

- 开发期间优先遵循官方文档、社区共识和框架最佳实践，不要为了眼前能跑就写临时拼凑方案。
- 遇到框架配置、部署配置、工具链配置或含糊需求时，先查询可靠来源再决定；不要靠猜测一路实现，最后跑偏再大改。
- 解决问题时要尽量定位根因，从更高一层的框架、架构和运行机制理解问题，而不是只修表面现象。
- 没有明确要求时，不要顺手添加额外功能、复杂兜底或无关优化。
- 如果存在多种理解且仓库内容无法判断，先向用户确认，再动手。

## 产品定位

- LLMSniffer 是给用户查看第三方 AI 中转站健康信息的监控列表，不是本站部署者自有服务的状态页，也不是账号池管理后台。
- 页面里的中转站都应被视为第三方站点；不要写成本站拥有、运维或直接提供这些中转站。
- 首页目标是帮助用户快速判断“哪个中转站现在更好用”：重点突出可用率、延迟体感、近期稳定性和趋势。
- 首页不应浪费大面积空间展示用户不关心的汇总卡片，例如正常数量、故障数量这类运维视角统计。
- 列表要高密度、易扫读，优先参考 RelayPulse 的可用性矩阵和 Uptime Kuma 的状态视觉；不要做成大字号、宽间距、老人模式 UI。
- 不要把 P50、P95 这类专业术语直接抛给普通用户；如果需要保留分位数据，应转译成“常规延迟”“高峰延迟”“慢请求”等用户能理解的表达。
- 不要在首页单独展示难以统一定义的“故障”列；失败和异常应尽量通过状态、可用率、趋势详情或 hover 信息表达。
- 首页列表默认不应横向滚动。字段过多时先删字段、合并字段或降低组件尺寸，而不是靠横向滚动硬塞。
- 收录天数、最后检测时间、复杂故障统计等低优先级信息不放主列表；必要时放进 hover、详情页或趋势格 tooltip。

## 本地开发约定

- 项目根目录就是当前目录，不要再创建一层 `llmsniffer/`。
- 开发期 Next.js 和 NestJS 都跑在宿主机；Docker 只启动 MongoDB、Redis 这类依赖服务。
- `npm run dev` 是开发入口：先执行 `docker-compose.dev.yml` 拉起 MongoDB/Redis，再并行启动 `apps/api` 和 `apps/web`。
- 单独启动依赖用 `npm run docker:dev`，停止开发依赖用 `npm run docker:dev:down`。
- 部署期使用 `docker-compose.prod.yml`，web、api、MongoDB、Redis 全部容器化；启动命令是 `npm run docker:prod`。
- 开发和生产 Compose 已用不同 `name` 隔离，不要合并成默认 `docker-compose.yml`，否则容易互相重建容器和卷。
- 临时截图、页面验证图片、调试图片等临时图片资源禁止写入项目根目录或任何已跟踪目录；需要保存时放到系统临时目录。

## Git 规范

- `main` 分支是正式分支，只用于合并已经完成审查的功能分支。
- 功能开发、问题修复和实验性改动必须在 `feature/*`、`fix/*` 或其他子分支上进行。
- 子分支不能直接合并进 `main`，必须通过 Pull Request 合并。
